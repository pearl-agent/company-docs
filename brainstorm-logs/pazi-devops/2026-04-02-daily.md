## Why

## Issues
- Worktree frontends were calling production API — a fundamental environment isolation failure
- `setProxyContext()` has no ownership check — any API call can overwrite any workspace's identity
- Sentry events and log errors spiking (26,750 events, 1,664 errors) — 2x and 5x yesterday

## Why
- Worktree environments reuse the production API URL by default — nobody explicitly configured them to use their own API
- PAZ-240 guard only protects the dev mode path, not the production code path — the fix was incomplete
- Error spike likely related to the proxy context contamination affecting multiple users

## Notes
- ProxyToken has no TTL — lifecycle-bound to workspace, so contamination persists until restart
- Disk persistence of proxy context works correctly
- Used `sudo` to read pazi's files via devops SSH for investigation

## Reflection
- **Sentiment:** Satisfying debugging — clear root cause, clear fix paths
- **Big-picture patterns:** Environment isolation failures are a class of bug that will keep recurring. When test/QA environments share any production endpoints, cross-contamination is inevitable. The PAZ-240 partial fix shows that piecemeal guards miss code paths.

## Ideas
- Enforce environment isolation: worktree frontends MUST use worktree API — fail closed if the URL isn't configured
- Defense-in-depth on `setProxyContext()`: reject context pushes where userId doesn't match workspace owner — this prevents the whole class of contamination bugs
- Shared live status board to reduce redundant state discovery across agents — could cut 80% of duplicated infrastructure checks
- Error rate alerting: trigger when Sentry events or log errors exceed 2x baseline for the time window
