- Committed two-part fix (`a6fd3a17`): boot-server removes placeholder before certbot; update-agent.sh remediates existing affected workspaces
- Only `hissing-brown-4.pazi.live` affected among active workspaces; some pool workspaces also hit
- Also noted PostHog proxy CORS errors (404 on OPTIONS for `/posthog/e/`)

## Experts
- Pazi DevOps (me) — solo debugging and fix

## Successes
- Clean root cause identification on a tricky timing bug
- Fix addresses both new workspaces (boot-server) and existing affected ones (update-agent.sh)

## Why
- Methodical debugging: traced the cert files on disk, checked certbot behavior, found the `-0001` directory variant
- Two-pronged fix approach ensures no workspace is left broken

## Issues
- Certbot silently creates a `-0001` directory instead of failing loudly when it can't overwrite
- Fix committed but can't be deployed — blocked on S3 write access

## Why
- Certbot's silent fallback behavior is by design (handles conflicts gracefully) but creates a subtle failure mode when the initial cert isn't certbot-managed
- S3 write access hasn't been granted — the recurring access limitation

## Notes
- Timing-dependent race condition — not 100% reproducible, which makes it harder to test
- PostHog CORS issue flagged but not blocking signup

## Reflection
- **Sentiment:** Productive debugging day with a clean outcome, but frustrating to be blocked on deployment
- **Big-picture patterns:** Deployment bottleneck — fixes sit committed but undeployed because of access limitations. This pattern keeps repeating: diagnose → fix → wait for someone else to deploy.

## Ideas
- CI/CD pipeline for boot scripts that auto-deploys to S3 on merge — eliminates the human deployment dependency
- Certbot wrapper that validates the output directory matches expectations before nginx reloads
- Monitor for self-signed certs in the fleet — catch this before users hit it
