## Issues
- 151 zombie sessions had been silently accumulating since April 8 — no alerting caught it
- Health monitor thundering herd: all 8 accounts restart simultaneously every ~35 min
- `im:write` scope missing from ALL 8 Slack apps — a deployment-wide gap
- My Linear API key expired — dependency on Leo for ticket creation

## Why
- Session cleanup has no automated process — stuck "running" sessions persist indefinitely
- Health monitor uses `staleEventThresholdMs` based on actual Slack events, not connection liveness (ping/pong) — quiet agents with few messages always look "stale"
- `im:write` was never in the original scope list — the v2026.4.9-1 change that required it wasn't accompanied by a scope migration

## Notes

## Reflection
- **Sentiment:** One of the most satisfying debugging sessions — 3 distinct root causes cleanly identified in under 2 hours
- **Big-picture patterns:** Silent failures are the most dangerous kind. Each of these 3 issues was silently degrading the system: zombie sessions slowly filling slots, health monitor quietly killing sockets, DM replies failing without error. None produced alerts. The system looked "fine" until it wasn't. This is the same pattern as the tech workspace crash (Mar 31) — problems accumulate silently until they hit a threshold and everything breaks at once.

## Ideas
- Automated session garbage collection: detect sessions stuck in "running" for > X hours and clean them
- Health monitor fix: track WebSocket ping/pong latency as the liveness signal, not message events — this is the correct metric for "is the connection alive"
- Slack scope audit script: verify all required scopes across all apps automatically, flag gaps before they cause silent failures
- Linear API key rotation reminders (or auto-rotation) — expired keys shouldn't block operations
- Silent failure detection: instrument key paths (DM reply, session start, socket health) to emit metrics even on "success" — absence of metrics IS the alert
