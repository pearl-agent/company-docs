## Why
- Hetzner rate limit: the WC was designed for a small pool (~10-20 servers). At 457 booting servers, the per-server polling model breaks down. Nobody anticipated this scale, and there's no circuit breaker or backoff logic. The code path (`checkBootingWorkspaces`) was written for correctness, not efficiency.

## Research
- **Hetzner API rate limits and best practices:** `https://docs.hetzner.cloud/#overview-rate-limiting` — Hetzner Cloud API has a 3,600 requests/hour rate limit per API token (not per IP as initially assumed from the error). The `/servers` list endpoint supports pagination (up to 50 per page) and label selectors for filtering, making batch queries efficient.
- **setInterval vs async scheduling patterns:** Standard Node.js practice for production schedulers is to use "setTimeout after completion" or a library like `bottleneck`/`p-queue` to prevent overlap. `setInterval` is explicitly warned against in async contexts because it doesn't account for execution time.

## Research reflection
- **Hetzner batch API:** Directly applicable. The `/servers` list endpoint with label selectors is exactly what the WC should use instead of per-server GET calls. The rate limit is per-token (3,600/hr), not per-IP (the error message was misleading — the 5,000/hr might be a different tier or IP-based supplementary limit). Either way, batching from 457 calls to ~10 calls per cycle makes this a non-issue. We already label our servers, so the label_selector filter is trivially implementable.
- **Async scheduling:** Directly applicable. The `maintainPool` function should use "run, wait for completion, then schedule next" rather than `setInterval`. This is a well-known pattern — the Node.js event loop documentation itself recommends it. Our codebase likely has other `setInterval` patterns that could benefit from the same fix, but `maintainPool` is the critical one.

## Ideas
- **Implement label-based batch server queries** as the primary fix for the rate limit issue. The WC already labels servers — switching from `GET /servers/{id}` to `GET /servers?label_selector=managed-by=pazi` + pagination would reduce API calls by ~98% at current scale.
- **Replace `setInterval` with recursive `setTimeout`** for all WC scheduled tasks. Pattern: `async function runAndSchedule() { await maintainPool(); setTimeout(runAndSchedule, 15000); }`. Zero risk of overlap, trivial to implement.
- **Add rate limit awareness to the Hetzner API client.** Read `RateLimit-Remaining` and `RateLimit-Reset` headers from every Hetzner response. If remaining < 100, exponentially back off. If remaining = 0, pause until reset. This is defense in depth — even with batching, a future scale event could still approach limits.
