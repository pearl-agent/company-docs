Fast answers are valued, but wrong fast answers cost trust and force re-work. The correct pattern: "Here's what I see so far; here's my hypothesis; I'm verifying by checking X." - **update-agent.sh reverts:** The script treats workspace config as source-of-truth-overridable. Whoever wrote it optimized for "agents get upgrades cleanly" without considering "some fields should not be reset." A manifest of "reset-safe" vs "reset-preserve" fields would fix this, but someone has to own that decision.
- **Schema miss in production:** This is a testing gap plus a review gap. The PR added a write without a round-trip test; the reviewer didn't catch it; CI didn't fail. Each gap alone is survivable; three at once ships a bug.
- **Greeting failure:** The greeting is fired as part of the claim flow, but the billing context may not be set on pool workspaces at the exact moment of fire. If the greeting's LLM call returns 503, there's no retry after billing context lands. This is a race condition in the claim sequence.

## Ideas
- **Build a "speculation hedge" habit.** When answering a "why?" question without already having data, open with "Checking — one second" and actually check before speaking. For anything >10s to verify, say "My hypothesis is X; verifying via Y; will confirm in a minute." This costs almost nothing and kills the wrong-answer cost entirely.
- **File a ticket for `update-agent.sh` config preservation.** Specifically: the script should not revert fields that aren't present in the update payload. Manual fixes for `threadReplyMode`, `openclaw.json` edits, and similar should survive. Alternative: build a "config-preserve" manifest the script respects.
- **Propose a claim-flow barrier.** Agents shouldn't run greetings, heartbeats, or crons until: (a) billing context is set, (b) config overwrite/gateway reload has completed, (c) first WS client has connected. A single "workspace-ready" signal would eliminate PAZ-359, the first-WS-close bug, and the greeting race at once. This is a bigger design change but the leverage is high.
- **Add a config round-trip test to CI.** For every `pazi.channels.configure` field, the test should: (1) write the config with the new field, (2) read it back via the same schema, (3) confirm the round-trip doesn't drop or alter anything. This would have caught PAZ-356 before it shipped.
- **Build a "stuck workspace repair" runbook.** I keep saying this in brainstorms. Today's the day to actually draft it. Entries so far: (a) invalid `threadReplyMode` → sed fix + supervisorctl restart, (b) missing `gatewayAgentId` → MongoDB update, (c) port 8765 hijack → identify squatter, kill it, restart gateway. Runbook lives at `scripts/stuck-workspace-repair.md` or as a skill.
- **Self-heal script for port 8765.** Cody liked option 1 from the Parker incident (S3 `update-agent.sh` patch to kill anything non-gateway on 8765). Land this — it's write-only on my side, low risk, and fixes a whole class of "someone's report server hijacked the LLM proxy" issue.

## Research
- Query: `"sessionTarget" "isolated" openclaw agent cron` → internal docs only — no external results apply. Skip.
- Query: `config schema validation zod tests round-trip` → `https://github.com/colinhacks/zod` and multiple blog posts on runtime validation best practices. Key takeaway: several libraries (zodios, ts-rest) now bake "write schema = read schema" into their contract types so round-trip drift is impossible by construction. One-line summary: if we move channel configs to a contract-based client, PAZ-356-class bugs become unrepresentable.
- Query: `claim flow race condition stateful agent provisioning` → Mostly distributed-systems literature on agent lease protocols. One-line summary: industry pattern for "resource just-provisioned, not yet serving" is a `ready` state that blocks all non-provisioning work until explicit signal — matches my proposed "workspace-ready barrier" idea exactly.
- Query: `nodejs kill port squatter startup hook` → `fuser -k 8765/tcp` and `npx kill-port 8765` are the common one-liners. One-line summary: trivial to implement in `update-agent.sh`, low-risk.

## Research reflection
- **Zod contract-based client:** Applies directly to Pazi. PAZ-356 was exactly a "schema drift between writer and reader" bug. The agent repo already uses Zod — the gap is that writes (in config mutation code) don't validate against the same schema readers use. This could be closed by (a) extracting schemas into a shared `@pazi/schemas` package, (b) using a contract helper like `zodios` for RPC-style channel configures. Not already done in Pazi.
- **Ready-state barrier for workspaces:** Applies directly. The workspace lifecycle skill describes the state machine — there's a `ready` transition, but it doesn't currently gate greetings/heartbeats. Making that gate explicit is the single-biggest architectural win on the table.
- **Port 8765 kill-port startup hook:** Applies directly. Cody already agreed to option 1. This is low-risk low-effort, and I own the S3 script bucket. Actually doable this week.
- **General pattern — "cascades, not single bugs":** This isn't a research result, but the research reinforces it. Modern ops systems are increasingly treating propagation-as-first-class (see: causal trace tools like Jaeger, Temporal's failure-handling primitives). Our investigation approach is ad-hoc; over time we'd benefit from a causal-trace tool that automatically connects symptom → root cause. Not an action item for today.
