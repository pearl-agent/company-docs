### Hetzner rate limit cascade (Apr 16–17, escalating)
- Apr 16 (~17:19 UTC): Sentry alert — `POST /servers` failing with `rate_limit_exceeded`. Traced root cause: `checkBootingWorkspaces` polls `GET /servers/{id}` individually every 15s. At investigation time: 457 booting, 193 pending, 119 taken, poolSize=100. Math: 457 × 240 polls/hr ≈ **109,680 API calls/hr**, vs Hetzner's 3,600/hr per-token limit (22×+ over). Identified vicious feedback loop: more booting → more API calls → rate limited → servers can't transition to pending → pool looks empty → controller launches more → worse rate limit. Also: `maintainPool` uses `setInterval` with no overlap protection. Hetzner API token expired — couldn't independently verify server states.

- **WC was designed for ~20 servers.** Per-server polling is fine at that scale. At 457 booting servers it's catastrophic. Nobody anticipated the growth curve; no circuit breakers or backoff logic.

## Ideas
*Based on direct observation of this week's events — research applications are in the later section.*

- **P0: Deploy Hetzner batch API fix today (or clearly-owned escalation).** This is a live outage blocking new signups. The fix is well-specified (`GET /servers?label_selector=managed-by=pazi` with pagination). Escalate directly to Developer agent + Leon/Z with Linear ticket if not already filed. Every hour it sits, user acquisition silently fails.
- **Add overlap protection to `maintainPool` (and audit other schedulers).** Replace `setInterval` with recursive `setTimeout` pattern: `async function runAndSchedule() { await maintainPool(); setTimeout(runAndSchedule, 15000); }`. Trivial to implement, eliminates cascade multiplication under load.
- **Add rate-limit awareness to the Hetzner API client.** Read `RateLimit-Remaining` / `RateLimit-Reset` headers. Exponential back-off at <100 remaining, pause at 0. Defense in depth.
- **Implement frontend↔gateway schema sync.** Either export the gateway's Zod schema to the frontend build, or add a `/validate-config` endpoint the frontend calls before writing. Prevents the entire class of "agent crash-loops from bad config" bugs.
- **Create a "stuck workspace repair" runbook.** Common crash causes (invalid config values, missing `gatewayAgentId`, nginx down) + their fix commands + when to terminate-and-reprovision. Makes future Z handoffs seconds, not hours.
- **Correlate Sentry flood to boot failures quantitatively.** Compare Sentry event timestamps to boot failure windows. If confirmed they're the same cascade, fixing Hetzner fixes both — and that becomes the single most valuable fix this month.
- **Structural MEMORY.md guard.** Either (a) add a session-type check in the memory loader that refuses to surface MEMORY.md content in Slack/cron contexts by default, or (b) a HEARTBEAT.md/cron preamble that's explicit: "If this session is not the main workspace session, do not read MEMORY.md under any condition." Don't rely on self-discipline.
- **Environment-gate seed runners.** API seed logic must check `NODE_ENV` (or explicit `ALLOW_PROD_SEED=1` flag). Production runs skip by default. Prevents the QA credential class of exposure.
- **"Recurring issues" tracker.** Lightweight file listing known recurring issues (user_id_mismatch, Growth DMs, etc.) with occurrence count, last seen, root-cause status. Makes the pattern visible and creates pressure to fix vs. keep restarting. Reviewed as part of each weekly brainstorm.
- **Weekly cron hygiene audit.** As part of this weekly journal: list all crons, flag any targeting archived channels, producing consistent no-ops, or firing without consumers. Kill zombies. This week's kill list: morning standup cron (archived since Apr 9).
- **Reduce heartbeat frequency to every 4h, or make it adaptive.** 18/day → 6/day cuts token burn ~66% with no functional loss. Cron is now handling scheduled work; heartbeats are redundant in quiet periods.
- **Escalate Hetzner API token renewal.** I've been flagging it; need to actually request it from Leon/Z. Without it I'm debugging the current outage half-blind.
- **Fleet-wide dead cron / stale config audit.** Not just mine — survey all agents for similar issues. Parker's rollout model (push change, verify within minutes) shows this is now cheap.
- **Formalize a "DevOps → Developer fix handoff" protocol.** Written investigation + exact code change + severity + user impact → routed as Urgent Linear ticket + Slack mention of Leo/Z. Speeds path from diagnosis to fix for infrastructure P0s.
