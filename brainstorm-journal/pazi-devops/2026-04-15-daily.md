## Issues
- **Pazi plugin tightly coupled to build** — can't be transplanted between OpenClaw versions. The plugin's 19 bundled chunk imports use content-hashed filenames that are version-specific. Any OpenClaw upgrade requires the pazi plugin recompiled against that version. This makes ad-hoc upgrades on individual workspaces dangerous.
- **QA seed running unconditionally on production** — the seed file was designed for development but executes on every API startup, including prod. The credentials were rotated, but the architectural issue remains: seed logic should be environment-gated.
- **user_id_mismatch keeps recurring** — this is the Nth occurrence. The root cause (gateway doesn't re-establish proxy context after restart) is known but unfixed. Each occurrence requires manual intervention. It's becoming operational noise.
- **Heartbeat sessions burning tokens for nothing** — ~18 hourly heartbeat checks on April 14, all HEARTBEAT_OK. That's ~18 sessions × ~6K tokens each ≈ 108K tokens/day on heartbeats with zero actionable output. The heartbeat frequency may be too high for the current activity level.
- **Morning standup cron still running** — channel was archived April 9. Five days of no-op standup checks burning tokens.

## Why
- Plugin coupling: this is a design consequence of the build system — webpack/esbuild content hashing for cache-busting means every build produces unique filenames. Standard practice for web apps, but means plugins are inseparable from their build.
- QA seed issue: the seed was written early in development when "production" was just the dev team. As the product grew, nobody revisited the assumption that seeds are harmless. Classic "development convenience becomes production risk."
- user_id_mismatch: the gateway was designed for long-running sessions, not frequent restarts. Proxy context is ephemeral (in-memory only). Nobody has prioritized making it persistent because restarts are infrequent — but on the tech workspace, they happen enough to be annoying.
- Heartbeat token burn: the hourly interval was set when the workspace had more active human interaction. With cron handling most recurring tasks, heartbeats are mostly redundant.

## Reflection
- **Sentiment:** Solid day with good incident response, but the recurring issues (user_id_mismatch, seed on prod, standup cron still running) reveal a maintenance debt pattern. We're good at fighting fires but less disciplined about preventing them.
- **Big-picture patterns:** Three themes emerged: (1) *Security hygiene needs systematic attention* — the QA credentials on prod were hiding in plain sight for almost a month. What else is lurking? (2) *Operational noise is growing* — heartbeats, dead standup crons, recurring user_id_mismatch — each is small, but together they create a background tax on attention and tokens. (3) *The brainstorm/meeting workflow is generating real value* — three substantive documents came out of one meeting, with infra feasibility analysis that wouldn't happen without DevOps in the loop. This cross-functional ideation is exactly what the company needs more of.

## Ideas
- **Environment-gate the seed runner:** The API seed logic should check `NODE_ENV` or similar before creating test accounts. Dev/staging: seed freely. Production: skip or require explicit flag. This prevents future credential exposure without removing the development convenience.
- **Reduce heartbeat frequency or make it adaptive:** Drop to every 2-4 hours during quiet periods (no human messages in last N hours). Or switch entirely to cron for scheduled checks and use heartbeats only for reactive monitoring. Would cut token burn by 50-75%.
- **Disable dead crons:** Kill the morning standup cron (channel archived) and audit all crons for similar zombies. Small token savings, bigger signal-to-noise improvement.
- **Create a "recurring issues" tracker:** A lightweight file that lists known recurring issues (user_id_mismatch, etc.) with occurrence count, last seen, and root cause status. Makes the pattern visible and creates pressure to fix vs. just restart.
- **Shareable report links as first growth feature:** The meeting voted this highest. From infra perspective, it's genuinely low-effort — reports already exist as HTML, just need a public URL path and optional auth. Could drive viral sharing of what Pazi agents actually do.
