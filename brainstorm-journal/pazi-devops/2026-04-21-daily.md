## Research
- Query: `redis shared multi-environment staging production cross-contamination` → standard DBA lit + PagerDuty / SRE blog posts. Shared cache between environments is a known anti-pattern; recommended approach is environment-prefixed keys (`staging:user:<id>:*`, `prod:user:<id>:*`) or fully separate Redis instances. Multiple post-mortems reference this exact failure mode.
- Query: `slack socket mode recv-q watchdog auto-reconnect` → followup from Apr 20 research. Further confirmed: apps running long-lived Socket Mode connections commonly add a watchdog layer because Slack's built-in pings don't detect all zombie-socket conditions. Recommended interval: 30-60s health check, with TCP-level Recv-Q or Last-Received-Event-Age checks.
- Query: `cron alert state change not state presence implementation sqlite` → several reference implementations use SQLite to persist previous-window counts and compute deltas. ~50 lines of Python; no new libraries needed. Matches the state-change-filter idea exactly.
- Query: `sentry 30 day rolling window quota recovery after fix` → Sentry docs confirm rolling windows recover gradually as old events age out. For a 30d window with burst emission on day N, significant recovery visible ~day N+20 onward; full recovery at day N+30. So the PAZ-365 recovery *will not* be visible in the quota percentage for weeks — I need to measure daily emission rate to confirm the fix worked.
- Query: `staging environment isolation architecture SaaS` → general best practices: separate databases, separate caches, separate queues, shared identity provider is acceptable. Our current setup (shared Redis) is explicitly called out as the riskiest form of environment coupling in every article I skimmed.
