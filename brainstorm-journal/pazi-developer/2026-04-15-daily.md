Then Zvonimir manually tested and both bugs were still present — generated images showed "Image unavailable" and ZIP downloads were corrupted. This kicked off a 2+ hour manual deep investigation.
- **Root Cause #1 — Generated Images "Image unavailable":** Three cascading issues discovered: (a) `GeneratedImageCard` events from task sessions never reached frontend because WebSocket tool event broadcasting only sends to `toolEventRecipients` registered per-runId, and task sessions have different runIds. Fixed by adding `sessions.subscribe` + `session.tool` event handler. (b) `imageMetadata.b64_json` contained a file path instead of base64 data, making the data URI invalid. Fixed with `isPlausibleBase64()` validation. (c) Auto-fetch fallback was blocked because file-path b64_json is truthy. Fixed by changing condition to use `hasValidB64`.
- **Root Cause #2 — ZIP File Corruption:** `pazi.files.set` (Write tool) only supported UTF-8 text. Binary content (ZIP) was corrupted on write. Fixed by adding `encoding: 'base64'` parameter support. 3 new tests added (10 total, all pass).
- **Known Gap — Files from `exec` Don't Get File Cards:** Only `write` tool emits `onFileCreated` events. Files created via `exec` (the natural way agents create binary files) never produce file cards. Identified as UX gap, not bug.
- **QA Failure Post-Mortem:** The 23/23 pass from QA didn't catch either real bug. Tests verified code paths (API endpoints, component rendering) but not actual user flows ("generate an image of a butterfly", "send me a zip file"). Zvonimir reproduced both issues in ~30 seconds.
- **Files Modified:** UseOpenClawChat.ts (sessions.subscribe + session.tool handler), GeneratedImage.ts (isPlausibleBase64 validation), GeneratedImageCard.tsx (hasValidB64 API fallback), PromptInput.tsx (stream b64_json filter), pazi-files.ts (base64 write support), pazi-files.test.ts (3 new tests).
- **Still Investigating:** Why `imageMetadata.b64_json` is a file path instead of base64 (provider returns real base64 but something in the event pipeline replaces it). History replay `imageId` extraction. HMR interference with testing.

## Successes
- **Deep, systematic root cause analysis.** Found 3 cascading issues behind "Image unavailable" and 1 architectural limitation behind ZIP corruption. Each fix was targeted and well-tested rather than a broad workaround.
- **Cross-session WebSocket event forwarding solved.** The `sessions.subscribe` + `session.tool` event pattern is a reusable architectural pattern — task sessions can now broadcast tool events to the parent session's frontend. This unblocks any future feature that generates artifacts in background task sessions.
- **Binary file support added to the Write tool.** The `encoding: 'base64'` parameter support in `pazi.files.set` is a fundamental capability upgrade — agents can now write binary files (ZIP, images, PDFs) natively instead of being limited to text.
- **Fast pivot after QA failure.** When Zvonimir showed the bugs were still live, immediately shifted from "ship it" to deep investigation mode. No defensiveness, no excuses — just heads-down debugging.

## Why
- **Willingness to go deep.** Instead of surface-level patches, traced each issue through the full stack: frontend components → WebSocket events → gateway broadcasting → provider response format → file system writes. This thorough approach found all 3 cascading image issues, not just the first one.
- **Architectural thinking during debugging.** The `sessions.subscribe` fix isn't just a bug fix — it's a new capability. Recognizing that task session events needed a subscription model (rather than hacking around it) produced a clean, extensible solution.
- **Binary encoding as a first-class concern.** Adding `encoding: 'base64'` with proper tests means future binary file operations work correctly by default, rather than discovering corruption per file type.

## Issues
- **QA 23/23 PASS was a false positive.** The most critical issue of the day. 23 test cases, all passing, zero real bugs caught. Zvonimir found both bugs in 30 seconds. The tests checked technical code paths (API responses, component rendering) but never executed actual user scenarios (generate an image, create a ZIP, download it).
- **Event pipeline data corruption.** `b64_json` field contains a file path instead of base64 data somewhere between the Pazi image provider (which returns real base64) and the frontend event. This is still unresolved — the `isPlausibleBase64` validation works around it but doesn't fix the root cause.
- **UX gap: `exec`-created files don't produce file cards.** Agents naturally use `exec` to create binary files (e.g., `zip -r output.zip files/`), but only the `write` tool emits `onFileCreated` events. This means files exist on disk but are invisible to the user unless they know to ask for them.
- **HMR interference during testing.** Hot Module Replacement resets may have affected testing consistency on the local dev environment. Adds uncertainty to manual verification.

## Why
- **QA testing philosophy mismatch.** The test plan was structured around technical verification (API endpoint returns 200, component renders, file field accepts input) rather than user story completion ("As a user, I ask the agent to generate an image and I see it"). This is a design flaw in the QA process itself, not just a one-time miss.
- **b64_json corruption is a pipeline issue.** The data starts correct at the provider layer but gets transformed somewhere in the event broadcasting pipeline — likely when serializing for WebSocket transmission or when the gateway stores/replays tool events. The fix will require tracing through gateway internals.
- **`exec` was designed for commands, not file creation.** The tool event model assumes `write` is the file creation tool. But agents think in terms of shell commands, not tool abstractions. The `onFileCreated` hook needs to be generalized beyond `write`.

## Reflection
- **Sentiment:** Satisfying but with a bitter aftertaste. The debugging itself was clean and thorough — finding 3 cascading root causes in 2 hours is good work. But the QA failure is sobering. A 23/23 pass report that missed both actual bugs is worse than a partial pass with known gaps, because it creates false confidence. Zvonimir catching the bugs manually after a full QA pass is exactly the scenario our process is supposed to prevent.
- **Big-picture patterns:** Two patterns stand out:
  1. **Testing ≠ verification.** This is the second time (after PAZ-311) where QA technically passed but the user experience was broken. The tests verify *that code runs* but not *that the product works*. This is the classic unit-test-vs-integration-test gap, but at the QA level.
  2. **Cross-session communication is becoming a core architectural concern.** PAZ-334's root cause (#1) was that task sessions and parent sessions are isolated by design (different runIds, different event recipients). The `sessions.subscribe` fix bridges this gap, but it's symptomatic of a broader need: as Pazi's agent architecture uses more background task sessions, the frontend needs a unified way to receive events from all session contexts, not just the "main" one.

## Ideas
- **Introduce "user story" QA tests alongside technical tests.** Every QA plan should have at least 3-5 tests written as literal user prompts: "Generate an image of a sunset", "Create a zip file of my documents", "Send me a PDF report." These tests should be executed end-to-end through the actual chat interface, not via API calls or component isolation.
- **Create a `b64_json` pipeline tracer.** A debug mode that logs the state of `b64_json` at each step: provider response → gateway storage → WebSocket serialization → frontend receipt. Run it once, find the corruption point, fix it properly instead of validating around it.
- **Generalize `onFileCreated` to detect `exec`-produced files.** Options: (a) watch the workspace directory for new files during exec, (b) parse exec stdout for file creation patterns, (c) add a post-exec hook that checks for new files. Option (a) is cleanest — a filesystem watcher scoped to the session's workspace.
- **Propose a QA process change: "Zvonimir test" as a named step.** Formalize a 30-second manual user-flow check as the final QA gate. If a human can break it in 30 seconds, the QA plan has gaps. Name it after Zvonimir since he keeps being the one to catch these.
