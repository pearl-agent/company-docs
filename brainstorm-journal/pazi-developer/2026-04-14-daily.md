## Research reflection
- **OAuth redirect URI limitations:** Directly applies. OpenAI's strict `localhost:1455` restriction is exactly the kind of redirect URI limitation the industry discussion is about. Our popup+paste approach is a pragmatic workaround within these constraints. A temporary localhost listener (our own local server on :1455) would be the ideal solution — this is common practice in CLI tools (e.g., `gcloud auth login` does exactly this).
- **Slack thread persistence:** Confirms our approach is correct — thread participation must be persisted at the application level, not through Slack's API. The existing `startSlackThreadCachePersistence` handles sent-cache but we need to extend it (or create a parallel mechanism) for participation-cache. This is a real gap that affects user experience.
- **Parallel dev environments (AirOps/Upsun):** Very relevant to our local-env struggles. AirOps' approach (per-worktree ports + DB + URL + local DNS) is exactly what `pazi-local-setup` attempts but with more polish. The Upsun article's warning about database race conditions is something we already handle (isolated DBs per feature env). However, the "shared Docker daemon" issue could explain some of our hostname/port conflicts. We should look at whether `pazi-local-setup` properly isolates Docker networks per feature environment.

## Ideas
- **Port the local HTTP listener pattern for Codex OAuth.** Tools like `gcloud`, `gh`, and VS Code all use temporary localhost listeners for OAuth callbacks. We could do the same: when user clicks "Connect Codex," the gateway spawns a temporary HTTP server on `localhost:1455`, opens the OAuth popup, captures the redirect, extracts the code, and exchanges it. No paste step needed. This directly improves UX polished-ness — a company goal.
- **Extend thread cache persistence to cover participation.** Fork the pattern from `startSlackThreadCachePersistence` to also persist/hydrate the participation cache. This is a straightforward extension — the infrastructure is already there, it just needs to be wired to the participation cache alongside the sent cache.
- **Adopt AirOps' local DNS routing pattern.** Instead of relying on port numbers for local env isolation, use `*.localhost` subdomains (e.g., `feature-name.localhost`). This eliminates cookie collisions (each subdomain gets its own cookies) and makes the env URLs human-readable. Could be a significant QA experience improvement.
- **Build a `pazi-local-setup doctor` command** inspired by `brew doctor` — runs a comprehensive check of the local environment and reports issues with suggested fixes. This is lower effort than fixing all 6+ bugs individually and immediately helps the team.
