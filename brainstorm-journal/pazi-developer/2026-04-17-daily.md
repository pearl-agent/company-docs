### Model aliasing / version pinning patterns in LLM APIs
- **Query:** how OpenAI and Anthropic handle model version aliases and deprecations
- **URL:** https://platform.openai.com/docs/models/how-we-use-your-data (and Anthropic's models docs)
- **Summary:** OpenAI uses versioned IDs (gpt-4o-2024-08-06) plus a floating alias (gpt-4o). Anthropic uses floating IDs (claude-3-5-sonnet-latest) with dated snapshots. Both providers pre-announce deprecations. Our rewrite layer is the Pazi-side equivalent — we're inserting an aliasing seam that the upstream providers expose natively. Worth knowing when upstream formalizes this: we can map our aliases to their `*-latest` IDs.

### Feature flag–gated model rollouts
- **URL:** https://launchdarkly.com/blog/llm-feature-flags/
- **Summary:** LaunchDarkly advocates feature-flagging model selection for gradual rollout, canary, and instant rollback. Our current rewrite is a hard flip — everyone moves from 4.6 to 4.7 at deploy time. For a same-price same-behavior point upgrade that's fine; for bigger model changes we'd want %-based rollout.

## Research reflection
- **Model aliasing:** The insight is that we should resist inventing our own alias semantics that diverge from upstream. If Anthropic ships `claude-opus-latest`, we should point our alias at that (or honor it pass-through) rather than maintain a parallel redirection map. The rewrite layer is temporary scaffolding until we formalize a deprecation→alias lifecycle. We should *not* build more of it without a design.
- **Feature-flagged rollouts:** Not needed for PAZ-357 (same pricing, same provider order, identical cache semantics). Will be needed for cross-family rewrites (e.g., if we ever wanted to map `claude-opus-*` → `gpt-5.4` for cost reasons, which would be crazy but illustrative). Note for future tickets: if a rewrite target has different pricing, the rewrite must be gated and observable, not a silent config flip.

## Ideas
- **Honor upstream aliases before inventing our own.** Add a unit test that fails if we add a Pazi alias when upstream already provides the same alias natively (`*-latest` on Anthropic). Force intentional divergence.
- **Rewrite observability baseline.** Add a Prometheus counter `pazi_model_rewrites_total{from, to}` in Proxy.ts. Today we only log structured events. A counter makes rollout dashboards trivial and gives us a signal when rewrites stop firing (i.e., agents have all updated their config).
- **Canary the next cross-family rewrite.** Before the next rewrite that crosses pricing tiers or providers, wrap the resolveModel call in a percentage gate (hash-based per userId) so we can stage rollout. Skip for identical-config rewrites like today's.
