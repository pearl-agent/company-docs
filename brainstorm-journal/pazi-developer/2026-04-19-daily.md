## Research

- **Rebase vs merge on long-lived feature branches.**
  Query: *"git rebase vs merge long-lived feature branch semantic conflicts strategy"*.
  URL: https://stackoverflow.com/questions/804115/when-do-you-use-git-rebase-instead-of-git-merge
  Summary: Industry consensus — rebase replays conflicts *per commit*, merge presents them *once*. For branches significantly behind main with semantic (API-shape) conflicts, a single merge is usually strictly less painful than an N-commit rebase. Matches today's conflict-#11 pivot.

- **Scope creep handling on bug tickets.**
  Query: *"large PR scope creep review cycles when to add defensive fix vs followup ticket"*.
  URL: https://www.reddit.com/r/ExperiencedDevs/comments/og3ife/whats_the_correct_way_to_deal_with_scope_creep_in/
  Summary: Experienced-dev consensus: when QA or a TL finds *adjacent* bugs on a PR that's already through multiple review cycles, the default should be **file a new ticket, don't expand the current PR**. Expanding a QA'd PR has real cost (re-review, re-QA, thread churn). Exception: if the new bug is a *provable, reproducible, state-dependent* fault in the same code path, a same-commit fix can be warranted. Matches Alan's Act II call *and* Alan's Act IV reversal once Ed's deterministic repro appeared.

- **Shared-branch conflict-avoidance practice.**
  Query (same search): *best practices r/git*.
  URL: https://www.reddit.com/r/git/comments/1autxuy/what_are_some_best_practices_when_dealing_with/
  Summary: Mainstream advice is "keep branches short-lived, rebase on target before merging." We already do short-lived branches in theory; in practice PAZ-376 was 2 h+ behind staging by the time the merge came up because the branch did a lot of implementation work during a busy staging window. Not a policy failure, a velocity-coincidence.
