## Why

Each round was a timing fix; the real fix was an architectural change. I should have asked "why does the state get lost?" instead of "when does the state get lost?"
- **Silent webchat failure** — OpenClaw's internal/external channel distinction isn't documented prominently. `webchat` exists as `INTERNAL_MESSAGE_CHANNEL` but isn't deliverable — this is a design choice (dashboard delivery uses session-queued events, not channel push), but the naming is misleading.
- **Extension skills in local env** — `pazi-local-setup` creates isolated Docker containers that only mount the workspace. Extension skills at `/app/extensions/pazi/skills/` are outside the workspace root, so the skill loader rejects them. This is an infrastructure gap, not a configuration error.
- **QA plan gaps** — Pre-implementation QA tests are designed from the ticket description, not from runtime behavior. The QA agent doesn't know about WebSocket event timing or OpenClaw's channel architecture. These are emergent integration issues that only surface in end-to-end testing.

### useSyncExternalStore for external state management
- **Query:** React useSyncExternalStore pattern for UI state surviving external re-renders SWR WebSocket
- **URL:** https://blog.logrocket.com/exploring-usesyncexternalstore-react-hook/
- **Summary:** LogRocket deep dive confirming useSyncExternalStore as the ideal hook for syncing React with external stores — exactly the pattern we used for UseTriggeredState.ts.

### React docs on useSyncExternalStore
- **URL:** https://react.dev/reference/react/useSyncExternalStore
- **Summary:** Official React documentation. Hook designed for subscribing to external stores outside React's lifecycle — validates our approach of module-level state that notifies React on change.

### WebSocket message handling patterns in React
- **URL:** https://websocket.org/guides/frameworks/react/
- **Summary:** Recommends useSyncExternalStore for high-frequency WebSocket updates where only relevant subscribers re-render. Suggests buffering for medium frequency (1-50 msg/s) and DOM-direct writes for high frequency.

### OpenClaw cron delivery documentation
- **URL:** https://docs.openclaw.ai/automation/cron-jobs
- **Summary:** Confirms "Delivery target missing/invalid means outbound was skipped" behavior. Documents that silent token suppresses both direct and fallback delivery paths.

## Research reflection
- **useSyncExternalStore (LogRocket + React docs):** Directly applicable — we already implemented this. The LogRocket article's "niche use cases" framing undersells it for our codebase. We have SWR + WebSocket + gateway events all triggering re-renders constantly. This should be our default for any transient UI state that isn't derived from server data. We should audit existing `useState` calls in components that also use SWR hooks.
- **WebSocket patterns (websocket.org):** The frequency-based guidance is useful context. Our WebSocket traffic is moderate (state updates every few seconds), so useSyncExternalStore is the right tier. We're not at the "bypass React entirely" level. The "only notify subscribers whose data changed" pattern could be useful if we ever build a shared event store.
- **OpenClaw cron delivery docs:** We discovered this the hard way. The documentation exists but we weren't looking at it because we assumed `webchat` was a valid channel. The lesson is to check OpenClaw docs *before* assuming channel behavior, not after debugging silent failures. This is now in MEMORY.md.
